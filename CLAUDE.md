# foot-tactic プロジェクトルール

## プロジェクト概要
サッカー用戦術ボードアプリ「TACTICALista」(https://app.tacticalista.com/) のクローンに、以下の独自機能を追加したWebアプリケーション。

- 戦術ボード下部のメモ機能(ユーザーが任意の見出し/タグを付与可能)
- AI音声文字起こしによるメモ入力(ユーザー自身のAIアカウントと紐付けて使用)
- 戦術ボード(スクショ画像)+メモのGoogleドキュメント保存(ログイン中のGoogleアカウントのドキュメントへ保存)

## 技術スタック
- フロントエンド: React
- バックエンド/インフラ: Firebase(Auth・Firestore・Hosting等)
- 認証: Googleアカウントログイン必須

## ⚠️ 最重要ルール: 不明点ゼロでの作業徹底
**このアプリの開発において、不明点がある場合は必ずユーザーに質問し、不明点が一切ない状態にしてから作業を進めること。**

- 仕様・ユーザーフロー・ビジネスロジック・UI詳細に少しでも曖昧さがあれば、実装前に必ず質問する
- 推測で仕様を決めて実装を進めることを禁止する
- このフローは要件定義・設計・実装・テスト、および**公開後の修正・UI改修・デザイン変更を含む全フェーズ**で常に徹底する
- 特にUI・レイアウト・デザインの変更はユーザーの意図が結果を大きく左右するため、配置・構成・見た目の方向性に少しでも曖昧さがあれば、必ず質問して不明点をゼロにしてから着手すること

## ⚠️ タスク管理ルール
- **progress.txt** — AIで実装可能なタスク(Phase X.Y形式)。完了したら即座に [x] と日付を記入する
- **manual-work.txt** — 人間の作業が必要なタスク(Work X.Y形式)。各Workに期限(どのPhase開始までに必要か)を記載

### Phase完了時の停止ルール(必須・厳守)
**1つのPhaseの全タスクが終了したら、必ず一度作業を止めてユーザーに確認を求めること。**
- ユーザーから問題がないことの確認が取れてから次のPhaseに進む
- 確認を待たずに勝手に次のPhaseの作業を進めることを禁止する

### Gitコミットルール(必須・厳守)
**各タスク(Phase X.Y)が終了したら、必ずgitにコミットしpushすること。**
- コミット形式: `<type>: <description>`(feat, fix, refactor, docs, test, chore)
- タスク完了 → progress.txt更新 → コミット → push を1セットとする

### OpenAI連携機能の扱い
**OpenAI APIキーは用意できない。** OpenAI連携機能(Phase6.3)は実装のみ行い、
テスト作成・実キーでの動作検証は行わない(型チェックのみ通す)。
音声文字起こしの動作検証・テストは全てGoogle Gemini(Phase6.4)で行う。

### 修正作業のフロー(必須・厳守)
**実装中・確認中に見つかった問題はすぐに修正せず、以下のフローで扱うこと。**
1. 問題の共有を受けたら **problem.txt** に記録する(P-連番、状態管理)
2. 問題をもとにタスクを切り出し、progress.txtの **Phase 8(修正作業)** にPhase8.X形式で追加する
3. 修正作業は全ての実装(Phase 7まで)が終わったPhase 8でまとめて実施する
4. 修正完了時はproblem.txtの状態を「修正済み」に更新する

### Phase着手前のWorkチェック(必須)
**新しいPhaseのタスクに着手する前に、必ずmanual-work.txtを確認すること。**
そのPhaseを期限とする未完了のWorkがある場合は、**Phaseのタスクを進めず**、
「必要なWork(番号と内容)が終了していないため着手できない」旨をユーザーに報告して作業を止める。

例: Phase1.2に着手する前にWork1.1が未完了 → 着手せずWork1.1の完了を依頼する
