## 1. プロジェクト概要

### 1.1 目的
サッカー用戦術ボードWebアプリ「TACTICALista」(https://app.tacticalista.com/) のクローンをベースに、本家にない以下の3つの独自機能を追加したWebアプリケーションを開発する。

1. **メモ機能** — 戦術ボードを見ながら試合についてのメモを取れる
2. **AI音声文字起こしメモ** — 試合から目を離さずに音声でメモを取れる
3. **Googleドキュメント保存** — ボード(画像)とメモをGoogleドキュメントとして書き出せる

### 1.2 ターゲットユーザー
- サッカーの試合を観戦しながら戦術分析・記録を行いたい人
- 指導者、アナリスト、戦術ブロガー、熱心なファン

### 1.3 本家TACTICALista調査結果(クローン対象の把握)
公式サイト(tacticalista.com)・公式ドキュメント(docs.tacticalista.com)の調査による。

- Googleアカウントログイン必須(他の認証手段なし)、クラウド保存、PWA対応
- プロジェクト種別: ボード(静止画)/アニメーション。一覧はサムネイル+タグ検索・絞り込み・並び替え
- フィールド: 11人制/8人制/フットサル、8種類の向き・表示領域、背景/ライン/レーン(縦5分割)/ゾーン(横3分割)の色設定、16:9・9:16切替、ズーム・パン
- オブジェクト10種: プレイヤー/ボール/マーカー/ライン/円形/矩形/ポリゴン/ポリライン/テキスト/手書き
- プレイヤー: ホーム・アウェイ属性、色/背番号/選手名/体の向き(矢印・アーム)/利き足表示
- フォーメーション一括配置: システム選択で一括配置、選手名・背番号リスト(CSVペースト対応)、控え選手タブ
- シーン: 1プロジェクト内に複数シーン(切替/複製/追加/削除/並び替え/サムネイル)
- 書き出し: 高解像度PNG、GIFアニメーション
- 料金: Starter無料(3プロジェクト/5シーン/30オブジェクト等)、Basic月$1.99(制限緩和+ロゴ非表示等)
- その他: 3Dビュー、ミニマルモード

## 3. 機能要件

### 3.1 認証 (AUTH)
| ID | 要件 |
|----|------|
| AUTH-1 | アプリ利用にはGoogleアカウントでのログインを必須とする(Firebase Authentication / Googleプロバイダ) |
| AUTH-2 | 初回ログイン時にユーザードキュメントをFirestoreに作成する |
| AUTH-3 | ログアウト、アカウント削除(退会)ができる。退会時はユーザーデータを削除する |
| AUTH-4 | Googleドキュメント保存に必要なGoogle APIスコープ(Drive/Docs)は、保存機能の初回利用時に追加同意(インクリメンタル認可)を求める |

### 3.2 プロジェクト管理 (PRJ)
| ID | 要件 |
|----|------|
| PRJ-1 | プロジェクト(=1試合/1テーマの戦術ボード)の新規作成・複製・削除ができる |
| PRJ-2 | プロジェクト作成時に競技種別(11人制/8人制/フットサル)を選択できる |
| PRJ-3 | プロジェクト一覧をサムネイル付きで表示する。更新日時順で並び、タグによる絞り込み・タイトル検索ができる |
| PRJ-4 | プロジェクトにタイトルと複数のタグを設定できる |
| PRJ-5 | プロジェクトデータはFirestoreに保存し、複数端末から同一アカウントでアクセスできる |
| PRJ-6 | 編集内容は自動保存する(明示的な保存ボタンも設ける) |

### 3.3 戦術ボード — フィールド (FLD)
| ID | 要件 |
|----|------|
| FLD-1 | フィールドの向き・表示領域を8種類(全景横/全景縦/ハーフ×向き等)から選択できる。切替時は配置済みオブジェクトの位置・向きを追従させる |
| FLD-2 | フィールド背景色・ライン色を変更できる |
| FLD-3 | レーン(縦5分割)・ゾーン(横3分割)の補助表示を色・不透明度付きで設定できる(不透明度0%で非表示) |
| FLD-4 | ボードのアスペクト比を16:9 / 9:16で切替できる |
| FLD-5 | ズーム・パンで特定エリアを拡大表示できる |
| FLD-6 | ウィンドウサイズに応じたフィット表示(デスクトップ) |

### 3.4 戦術ボード — オブジェクト (OBJ)
| ID | 要件 |
|----|------|
| OBJ-1 | 次の10種類のオブジェクトを配置できる: プレイヤー/ボール/マーカー/ライン/円形/矩形/ポリゴン/ポリライン/テキスト/手書き |
| OBJ-2 | 図形類はドラッグで配置時サイズを指定できる。連続配置モードを持つ |
| OBJ-3 | 選択: クリック選択/ドラッグ矩形選択/Shift+クリック追加選択/種類別一括選択 |
| OBJ-4 | 削除: Delete・Backspaceキーおよび削除ボタン |
| OBJ-5 | 編集: 移動/リサイズ/回転/色変更/重なり順変更/複製(Ctrl・Cmd+D)/コピー&ペースト |
| OBJ-6 | 整列: 左揃え/右揃え/上揃え/下揃え/水平分布/垂直分布 |
| OBJ-7 | Undo/Redoに対応する |
| OBJ-8 | プレイヤー: ホーム/アウェイ属性、ボディカラー、背番号+色、選手名+色+表示位置、体の向き矢印/アーム表示、利き足表示。ボディサイズと選手名フォントサイズは全プレイヤー共通設定 |
| OBJ-9 | プレイヤーの向きは回転ハンドルで変更できる。「ボール方向を向く」補助ボタンを設ける |

### 3.5 フォーメーション一括配置 (FRM)
| ID | 要件 |
|----|------|
| FRM-1 | ホーム/アウェイ別にフォーメーション(4-4-2、4-3-3等の主要システム)を選択し、プレイヤーを一括配置できる |
| FRM-2 | 選手名・背番号のリスト入力に対応。CSVテキストのペーストで一括入力できる。背番号省略時はポジション名を表示する |
| FRM-3 | Line-up(先発)/Substitute(控え)のタブを持ち、控えはフィールド外に並べられる |
| FRM-4 | フィールド中央基準の配置オプションを持つ |

### 3.6 シーン (SCN)
| ID | 要件 |
|----|------|
| SCN-1 | 1プロジェクト内に複数シーンを作成できる(切替/空シーン追加/現在シーンの複製/削除/ドラッグで並び替え) |
| SCN-2 | シーンごとにサムネイルを表示する |

### 3.7 書き出し (EXP)
| ID | 要件 |
|----|------|
| EXP-1 | 表示中のシーンを高解像度PNG画像としてダウンロードできる |

### 3.8 【独自】メモ機能 (MEMO)
| ID | 要件 |
|----|------|
| MEMO-1 | 戦術ボード画面の**下部**にメモエリアを常設する(折りたたみ可能) |
| MEMO-2 | メモは**プロジェクト単位**で保持する。どのシーンを表示していても同じメモ一覧が表示される |
| MEMO-3 | メモは複数エントリの一覧形式。各エントリはテキスト本文+作成日時を持つ |
| MEMO-4 | 各エントリに**ユーザーが任意の見出し(タグ)**を付けられる(例:「フォーメーション」「決定機シーン」)。タグは自由入力+過去に使ったタグのサジェスト |
| MEMO-5 | タグでメモ一覧を絞り込みできる |
| MEMO-6 | エントリの編集・削除・並び(作成日時順)に対応する |
| MEMO-7 | メモはプロジェクトデータと同様Firestoreに自動保存する |

### 3.9 【独自】AI音声文字起こし (VOICE)
| ID | 要件 |
|----|------|
| VOICE-1 | メモエリアに**録音ボタン**を設置。押して発話→停止で、録音音声を文字起こしAPIに送信し、結果を新規メモエントリとして挿入する(1発話=1メモ) |
| VOICE-2 | 対応プロバイダ: **OpenAI**(gpt-4o-transcribe / Whisper)と**Google Gemini**。ユーザーが設定画面でプロバイダを選択する |
| VOICE-3 | ユーザーは**自分のAPIキー**を設定画面で登録する(=ユーザーのAIアカウントとの紐付け)。文字起こしの利用料はユーザー自身のAPIアカウントに課金される旨を画面に明示する |
| VOICE-4 | APIキーは**端末のlocalStorageにのみ保存**し、サーバー(Firebase)には送信・保存しない。この旨を設定画面に明示する |
| VOICE-5 | APIキー未登録で録音ボタンを押した場合、設定画面へ誘導する |
| VOICE-6 | 文字起こし結果は挿入前にプレビューし、そのまま採用/編集/破棄を選べる。挿入後のエントリにもタグを付けられる |
| VOICE-7 | マイク権限拒否・API認証エラー・ネットワークエラー時は日本語/英語でエラー内容と対処を表示する |
| VOICE-8 | 録音はブラウザのMediaRecorder APIで行い、クライアントから直接プロバイダAPIへ送信する(音声・キーともに自社サーバーを経由しない) |
| VOICE-9 | APIの無料枠(クォータ)超過エラーを検知した場合、「無料枠を使い切った」旨を通知し、以降のAPI呼び出しをブロックして録音機能を無効化する。ブロック状態はlocalStorageに保持し、ユーザーが手動で再有効化できる(クォータの日次リセットに対応) |

### 3.10 【独自】Googleドキュメント保存 (GDOC)
| ID | 要件 |
|----|------|
| GDOC-1 | ボード画面の「Googleドキュメントに保存」ボタンで、**ログイン中のGoogleアカウント**のGoogleドライブに**新規Googleドキュメントを作成**して保存する(保存のたびに新規作成) |
| GDOC-2 | ドキュメント内容: タイトル(プロジェクト名+保存日時)→**全シーンのスクリーンショット画像をシーン順に貼付**(各画像にシーン番号の見出し)→末尾にメモ一覧(タグ見出し+本文+日時)を書き込む |
| GDOC-3 | スクリーンショットはクライアント側でシーンをPNGレンダリングして生成する(EXP-1と同じ描画基盤を使用) |
| GDOC-4 | Google Docs API / Drive APIを使用。必要スコープは初回保存時にインクリメンタル認可で取得する(AUTH-4) |
| GDOC-5 | 保存完了時に作成されたドキュメントへのリンクを表示する |
| GDOC-6 | 認可エラー・API失敗時はリトライ可能なエラーメッセージを表示する |

## 5. 技術要件

### 5.1 技術スタック
| 層 | 技術 | 備考 |
|----|------|------|
| フロントエンド | React + TypeScript (Vite) | SPA |
| 描画 | Konva.js (react-konva) | Canvasベース。オブジェクト操作・PNG出力に適する |
| 状態管理 | Zustand | Undo/Redo実装を含む |
| i18n | react-i18next | 日本語/英語 |
| 認証 | Firebase Authentication (Googleプロバイダ) | |
| DB | Cloud Firestore | プロジェクト/シーン/メモ |
| ホスティング | Firebase Hosting | |
| 外部API | OpenAI Audio API / Gemini API(クライアント直接呼び出し)、Google Docs API / Drive API | |

### 5.2 データモデル(Firestore)
```
users/{uid}
  ├ displayName, email, createdAt
  ├ settings: { language, aiProvider }   // APIキーは保存しない
  └ plan: { type: "free", trialEndsAt: null }   // 将来課金用(§8)

users/{uid}/projects/{projectId}
  ├ title, tags[], sportType(11人制|8人制|フットサル)
  ├ fieldSettings: { layout, aspect, colors{bg,line,lane,zone}, ... }
  ├ thumbnailUrl, createdAt, updatedAt
  ├ scenes: [ { id, order, objects[] } ]   // objects: type別のプロパティを持つ
  └ (サブコレクション) memos/{memoId}: { text, tags[], source(manual|voice), createdAt, updatedAt }
```

### 5.3 セキュリティ要件
- Firestoreセキュリティルール: 自分のuid配下のデータのみ読み書き可
- AIのAPIキーはlocalStorageのみに保存し、Firebaseへ送信・保存・ログ出力しない
- Google OAuthトークンはFirebase Auth/ Google Identity Servicesの標準管理に従い、自前保存しない
- 全ユーザー入力(タイトル、タグ、メモ本文)のバリデーションを実施
- シークレットのハードコード禁止(環境変数で管理)

### 5.4 非機能要件
- 対応ブラウザ: Chrome最新(PC/Android)、Safari(iOS/iPadOS 16+)、Edge最新
- レスポンシブ対応(PC/タブレット/スマートフォン)。PWA対応(ホーム画面追加)
- ボード操作は60fpsを目標とし、オブジェクト100個程度まで快適に動作すること
- 自動保存はデバウンス(数秒)してFirestore書き込み回数を抑制する
- テスト: ユニット+統合+E2E、カバレッジ80%以上(グローバルルール準拠)

## 8. 将来の課金導入を見据えた設計前提

初期リリースは全機能無料だが、後日課金機能を導入する前提で以下の構造を最初から用意する。

- ユーザードキュメントに `plan` フィールド(type / trialEndsAt)を持たせる
- 機能制限チェックは `entitlements`(プラン→利用可能機能・上限のマッピング)を参照する一元化された仕組みとし、初期リリースでは全員「無制限」を返す
- 課金導入時の想定仕様: **登録から14日間は無料トライアル**(その後廃止の可能性あり)。決済手段は導入時に別途要件定義する
- プロジェクト数・シーン数・オブジェクト数の計測ロジックは初期から実装しておく(制限は掛けない)

## 9. 開発ルール(再掲)

**不明点がある場合は必ずユーザーに質問し、不明点が一切ない状態で作業を進めること。** 仕様・フロー・UIの曖昧さを推測で埋めて実装することを禁止する(CLAUDE.md参照)。
